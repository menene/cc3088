Modelo Entidad Relación (ER)

Semestre 1, 2025

Un diseño bien hecho desde el principio evita problemas en la implementación.

¿Qué es el Modelo ER?

El modelo ER es una herramienta fundamental para representar datos y relaciones de manera clara y estructurada (UML) antes de implementarlos en un DBMS.

Objetivo Principal

Ofrecer un diseño conceptual independiente del sistema de gestión de bases de datos que se utilizará.

Abstracción.

Historia del Modelo ER

El modelo Entidad-Relación fue introducido en 1976 por Peter Chen. Motivado a proporcionar una herramienta visual que simplificara la representación de datos complejos.

  • Adoptado en la academia y la industria.
  • Base para diseños conceptuales en bases de datos relacionales.

Componentes del Modelo ER

El modelo ER consta de tres elementos clave:

Entidades

Representan objetos o conceptos del mundo real, como Clientes o Productos.

Atributos

Propiedades o características de las entidades, como Nombre o Correo Electrónico.

Relaciones

Asociaciones entre entidades, como "Un cliente realiza un pedido".

Claves en el Modelo ER

Clave Primaria (PK)

Atributo único que identifica a cada registro en una entidad.

Clave Foránea (FK)

Relación entre entidades mediante un atributo que apunta a la clave primaria de otra entidad.

Clave Compuesta

Se forma con dos o más atributos. Ejemplo: En DetallePedido, la clave primaria es la combinación de pedido_id y producto_id.

Llaves Candidatas

Son atributos que podrían ser seleccionados como clave primaria porque identifican de manera única cada registro. Ayudan a identificar la clave primaria más adecuada.

Ejemplo: En una tabla de empleados, empleado_id y seguro_social son llaves candidatas.

Tipos de Relaciones

1:1

Uno a Uno

Cada instancia de una entidad está relacionada con exactamente una instancia de otra entidad.


Un estudiante tiene una carrera.

1:N

Uno a Múchos

Una instancia de una entidad puede estar relacionada con muchas instancias de otra entidad, pero no al revés.


Un estudiante tiene varias calificaciones.

N:M

Múchos a Múchos

Muchas instancias de una entidad pueden estar relacionadas con muchas instancias de otra entidad.


Un estudiante puede inscribirse en varios cursos, y cada curso tiene varios estudiantes inscritos.

Cardinalidad

La cardinalidad define el número de veces que una instancia de una entidad puede asociarse con instancias de otra entidad.

Importancia

  • Define las reglas de negocio (¿puede un cliente tener múltiples cuentas bancarias?).
  • Optimiza consultas y estructuras de datos.
  • Valida las relaciones entre entidades.

Diagramas ER

Los diagramas ER son representaciones gráficas que muestran las entidades, atributos y relaciones de un sistema. Sirven como una herramienta visual para diseñar y comunicar la estructura conceptual de una base de datos.

Importancia de Estandarizar

  • Facilita la comunicación entre equipos.
  • Aceleran el proceso de desarrollo.
  • Reduce errores en la implementación.
  • Mejora la mantenibilidad y escalabilidad.

Tipos de Atributos

Atributos Multivaluados

Representan valores múltiples para un atributo.

  • Ejemplo: Un cliente con varios números de teléfono.

En el diseño físico, los atributos multivaluados suelen representarse como una tabla separada.

  • Ejemplo: Una tabla cliente_telefonos podría tener cliente_id y numero_telefono.

Atributos Derivados

Calculados a partir de otros valores.

  • Ejemplo: Edad calculada a partir de la fecha de nacimiento.

Reducen la redundancia al no almacenar datos que pueden calcularse.

Se deben calcular dinámicamente en consultas para mantener los datos consistentes.

Tipos de Relaciones

Relaciones Fuertes

Las entidades tienen claves primarias propias.

  • Ejemplo: Cliente y Pedido.

Relaciones Débiles

Las entidades dependen de otras para identificarse.

  • Ejemplo: DetallePedido necesita asociarse a un Pedido.

Identificar relaciones fuertes y débiles permite estructurar el modelo ER de manera más precisa, priorizando la integridad referencial.

Jerarquías

Generalización

Combina entidades específicas en una entidad general.

  • Ejemplo: Carros y Motos se agrupan en Vehículos.
  • Los atributos generales, como marca y modelo, se colocan en Vehículos

Especialización

Divide una entidad general en subentidades específicas.

  • Ejemplo: Persona puede ser Cliente o Empleado.
  • Cliente podría tener atributos específicos como historial_compras.
  • Empleado podría tener atributos específicos como salario.

Estos conceptos ayudan a organizar la estructura jerárquica de las entidades y facilitan el manejo de atributos comunes y específicos.

Errores Comúnes

  • Redundancia de datos, lo que genera inconsistencias.
  • Relaciones mal definidas o ausentes.
  • Diseño que no escala bien para datos futuros.
